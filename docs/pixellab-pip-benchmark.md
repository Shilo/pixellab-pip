# PixelLab Pip Skill Benchmark

This benchmark measures what the PixelLab Pip skill costs an agent and how well it routes, compared with the other ways to give an agent PixelLab knowledge:

- **Pip** — the skill as it is now (this repo's working tree): `SKILL.md` injected, `references/*.md` read on demand.
- **Pip (Old v0.5.0)** — the skill before its size refactor ([tag `pre-kiss-yagni-refactor`, commit 6fdae41](https://github.com/Shilo/pixellab-pip/commit/6fdae41)), an unoptimized version kept for the efficiency-over-time view.
- **PixelLab MCP Docs** — no skill, but PixelLab's official [`api.pixellab.ai/mcp/docs`](https://api.pixellab.ai/mcp/docs) injected into the session (the [pixellab.ai/mcp](https://www.pixellab.ai/mcp) "include this link in your prompts" tip).
- **Vanilla** — no skill and no docs: the agent's own knowledge only.

It measures the **agent session only** (context tokens, output, cost, routing correctness) — never PixelLab credits. Everything here is produced by [`dev-tools/skill_benchmark.py`](../dev-tools/skill_benchmark.py) and is reproducible; see [Reproduce](#reproduce). Routing correctness is scored by deterministic regexes (no LLM judge), so the same response always scores the same.

The tables below are generated by the benchmark itself. **Context size** (chars/4 estimates) is deterministic for the Pip and git columns; the PixelLab MCP Docs column is fetched live from PixelLab's official docs, so it can drift as they update. ## Reading the results

- **Vanilla is cheapest but least reliable.** With no PixelLab knowledge it tends to pick the wrong tool and to miss safety refusals — low correctness for near-zero context.
- **PixelLab MCP Docs handles MCP tool selection** because it carries the full MCP inventory, but it misses the cases that need more than picking a hosted tool: REST-only routes, cost/ordering detail, local post-processing, and the skill's setup/fallback policies. Those show up as its 0-scores in the table.
- **Where Pip pulls ahead:** several scenarios come from the [showcase](showcase/README.md) and target exactly those gaps — complete icon/tile **sheets** route to REST `generate-image-v2` (not the MCP UI/tile tools a pure-MCP context reaches for), the tiles-vs-tileset distinction, 1-bit/palette recolor as **local** Aseprite processing, and the background-removal fallback. Pip routes them the way the showcase results did; the official docs alone do not carry that knowledge.
- **Pip (the skill)** routes correctly across the benchmark scenarios. Its **up-front** context is close to the official docs (`SKILL.md` ~7.2k vs PixelLab MCP Docs ~7.7k), but on tasks that need a reference it reads one on demand and spends **more** session tokens than the flat docs injection — that reference load is what buys the correct, documented answer. Simple routing that needs no reference stays comparable to or leaner than the docs.
- **The refactor** cut the base `SKILL.md` context roughly a third versus Pip (Old v0.5.0), visible in the context table.

The two context strategies (skill vs official docs) are complementary, not rivals — the skill itself treats the official docs as the authoritative MCP inventory; see [Official MCP Docs vs Pip Skill](tools/pixellab-mcp-docs-vs-pip-skill.md).

## Reproduce

No agent CLIs or secret needed (the PixelLab MCP Docs arm still does one live fetch of the official docs). The Pip and git columns are deterministic; regenerate the **context size** table with:

```powershell
python dev-tools/skill_benchmark.py --static --variants pre-kiss-yagni-refactor,mcp-docs,vanilla
```

Live routing (needs an agent CLI; numbers vary run to run). This writes results to `.local/bench/<stamp>/` — read its `SUMMARY.md`:

```powershell
python dev-tools/skill_benchmark.py --agents claude --variants pre-kiss-yagni-refactor,mcp-docs,vanilla `
  --scenarios route-hex-tiles,route-character,cheap-animation,refuse-internal-endpoint --reps 2
```

To refresh the tables in *this* report, run the **full** scenario set (no `--scenarios` filter) with `--report docs/pixellab-pip-benchmark.md`, or use the `full` preset below — a partial `--scenarios` run with `--report` would shrink the published routing table to only those rows.

On Windows, [`dev-tools/run-skill-benchmark.ps1`](../dev-tools/run-skill-benchmark.ps1) wraps the common runs. Its **`full`** preset runs every agent and every variant, live + paid, and refreshes this report automatically. Full methodology, arms, and flags are in [Developer → Skill Token Benchmark](developer.md#skill-token-benchmark).

## Method and caveats

- **Agent session only.** Tokens/cost are the agent's, not PixelLab credits.
- **Static tokens are chars/4 estimates**, not a tokenizer count; treat them as relative, not absolute.
- **The routing snapshot is nondeterministic.** Agent runs vary; the context table is the reproducible part. Rerun with more agents/reps (or the `full` preset) for your own numbers. The generation note above records when it was last measured and with which agents/reps.
- **Fairness controls:** each variant runs in an isolated workspace outside the repo (so no stray `CLAUDE.md`/`AGENTS.md` leaks in), variants interleave within each rep, medians are reported, and routing is scored by fixed regexes rather than a model.
- **The PixelLab MCP Docs content is fetched live** from `api.pixellab.ai/mcp/docs` at run time and saved alongside the results for audit.
