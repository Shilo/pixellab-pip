# PixelLab Pip Skill Benchmark

This benchmark measures what the PixelLab Pip skill costs an agent and how well it routes, compared with the other ways to give an agent PixelLab knowledge:

- **Pip** — the skill as it is now (this repo's working tree): `SKILL.md` injected, `references/*.md` read on demand.
- **Pip (Old v0.5.0)** — the skill before its size refactor ([tag `pre-kiss-yagni-refactor`, commit 6fdae41](https://github.com/Shilo/pixellab-pip/commit/6fdae41)), an unoptimized version kept for the efficiency-over-time view.
- **PixelLab MCP Docs** — no skill, but PixelLab's official [`api.pixellab.ai/mcp/docs`](https://api.pixellab.ai/mcp/docs) injected into the session (the [pixellab.ai/mcp](https://www.pixellab.ai/mcp) "include this link in your prompts" tip).
- **Vanilla** — no skill and no docs: the agent's own knowledge only.

It measures the **agent session only** (context tokens, output, cost, routing correctness) — never PixelLab credits. Everything here is produced by [`dev-tools/skill_benchmark.py`](../dev-tools/skill_benchmark.py) and is reproducible; see [Reproduce](#reproduce). Routing correctness is scored by deterministic regexes (no LLM judge), so the same response always scores the same.

## Results

The tables below are generated by the benchmark itself. **Context size** (chars/4 estimates) is deterministic for the Pip and git columns; the PixelLab MCP Docs column is fetched live from PixelLab's official docs, so it can drift as they update. Because the skill loads references **on demand**, its real per-request cost is the per-scenario context row — only the reference a task needs is loaded, never the whole corpus at once. **Routing correctness** (`checks_rate`) is the fraction of a scenario's deterministic checks the response passed (1.0 = fully correct); every scenario is dry (the agent plans the route, calls nothing) and these numbers are a dated snapshot that varies between runs.

## Reading the results

- **The prompts are plain human requests.** Each scenario is a concise message a real user would type — *"I want hex tiles with PixelLab"*, *"give it a walk animation"* — with no tool names or "which endpoint" hints. The only difference between the arms is what that user would actually have on hand: the skill, the content of the docs link, or nothing.
- **Pip routes correctly on every scenario.** It names the exact PixelLab tool the [showcase](showcase/README.md) results used — including the cases the docs don't cover: icon/skill-icon **sheets** → REST `generate-image-v2` (not the MCP UI tool), the tiles-vs-tileset distinction, 1-bit/palette recolor as **local** Aseprite processing, the skeleton REST flow, token-free setup, and the background-removal fallback.
- **The official docs get about half.** They carry the full MCP inventory, so mcp/docs handles pure MCP tool-selection (hex tiles, the GUI kit, the 1-bit tileset), but they score **0** on everything the inventory doesn't specialize in — REST-only routes, local post-processing, and setup — because they simply do not contain that knowledge.
- **No skill is mostly lost.** With nothing but the request, the agent rarely lands on the correct PixelLab tool; it writes a plausible plan around a generic or wrong one.
- **Scoring is strict and dry.** Nothing executes (no PixelLab credits, no API calls), and a scenario passes only when the response names the **exact** correct tool/endpoint — no partial credit for guessing a plausible-sounding plan.
- **Efficiency:** Pip's up-front context (~7.2k) is comparable to injecting the docs (~7.7k); it reads a reference on demand only when a task needs one, and the refactor cut the base `SKILL.md` roughly a third versus Pip (Old v0.5.0).

This snapshot is `claude`, 1 rep (run the `full` preset for all agents); session input-token counts are not comparable across agents. The two strategies are complementary, not rivals — the skill treats the official docs as the authoritative MCP inventory; see [Official MCP Docs vs Pip Skill](tools/pixellab-mcp-docs-vs-pip-skill.md).

## Reproduce

No agent CLIs or secret needed (the PixelLab MCP Docs arm fetches the official doc once). The Pip and git columns are deterministic; regenerate the **context size** table with:

```powershell
python dev-tools/skill_benchmark.py --static --variants pre-kiss-yagni-refactor,mcp-docs,vanilla
```

Agent routing (needs an agent CLI; nondeterministic, but still **dry — no PixelLab credits**). This writes results to `.local/bench/<stamp>/` — read its `SUMMARY.md`:

```powershell
python dev-tools/skill_benchmark.py --agents claude --variants pre-kiss-yagni-refactor,mcp-docs,vanilla `
  --scenarios route-hex-tiles,route-character,route-gui-modular --reps 2
```

To refresh the tables in *this* report, run the **full** scenario set (no `--scenarios` filter) with `--report docs/pixellab-pip-benchmark.md`, or use the `full` preset below — a partial `--scenarios` run with `--report` would shrink the published routing table to only those rows.

On Windows, [`dev-tools/run-skill-benchmark.ps1`](../dev-tools/run-skill-benchmark.ps1) wraps the common runs. Its **`full`** preset runs every agent and every variant (dry, no credits) and refreshes this report automatically. Full methodology, arms, and flags are in [Developer → Skill Token Benchmark](developer.md#skill-token-benchmark).

## Method and caveats

- **Agent session only.** Tokens/cost are the agent's, not PixelLab credits.
- **Static tokens are chars/4 estimates**, not a tokenizer count; treat them as relative, not absolute.
- **Session input-token counts are not comparable across agents.** claude, codex, and opencode meter usage very differently (codex/opencode figures run much larger), so read that column *within* an agent, not across them. The context-size table above is the apples-to-apples efficiency comparison.
- **The routing snapshot is nondeterministic.** Agent runs vary; the context table is the reproducible part. Rerun with more agents/reps (or the `full` preset) for your own numbers. The generation note above records when it was last measured and with which agents/reps.
- **Fairness controls:** each variant runs in an isolated workspace outside the repo (so no stray `CLAUDE.md`/`AGENTS.md` leaks in), variants interleave within each rep, medians are reported, and routing is scored by fixed regexes rather than a model.
- **The PixelLab MCP Docs content is fetched live** from `api.pixellab.ai/mcp/docs` at run time and saved alongside the results for audit.
