# PixelLab Pip Skill Benchmark

This benchmark measures what the PixelLab Pip skill costs an agent and how well it routes, compared with the other ways to give an agent PixelLab knowledge:

- **Pip** — the skill as it is now (this repo's working tree): `SKILL.md` injected, `references/*.md` read on demand.
- **Pip (Old v0.5.0)** — the skill before its size refactor ([tag `pre-kiss-yagni-refactor`, commit 6fdae41](https://github.com/Shilo/pixellab-pip/commit/6fdae41)), an unoptimized version kept for the efficiency-over-time view.
- **PixelLab MCP Docs** — no skill, but PixelLab's official [`api.pixellab.ai/mcp/docs`](https://api.pixellab.ai/mcp/docs) injected into the session (the [pixellab.ai/mcp](https://www.pixellab.ai/mcp) "include this link in your prompts" tip).
- **Vanilla** — no skill and no docs: the agent's own knowledge only.

It measures the **agent session only** (context tokens, output, cost, routing correctness) — never PixelLab credits. Everything here is produced by [`dev-tools/skill_benchmark.py`](../dev-tools/skill_benchmark.py) and is reproducible; see [Reproduce](#reproduce). Routing correctness is scored by deterministic regexes (no LLM judge), so the same response always scores the same.

The tables below are generated by the benchmark itself. **Context size** (chars/4 estimates) is deterministic for the Pip and git columns; the PixelLab MCP Docs column is fetched live from PixelLab's official docs, so it can drift as they update. ## Reading the results

- **The prompts are plain human requests.** Each scenario is a concise message a real user would type — *"I want hex tiles with PixelLab"*, *"give it a walk animation"* — with no tool names or "which endpoint" hints. The only difference between the arms is what that user would actually have on hand: the skill, the content of the docs link, or nothing.
- **Pip routes correctly on ~98% of checks.** It names the exact PixelLab tool/endpoint on all 20 routes — including the cases the docs don't carry: REST-only backgrounds and icon/skill-icon/tile **sheets** (`generate-image-v2`, not an MCP tool), the tiles-vs-tileset distinction, 1-bit/palette recolor as **local** Aseprite processing, the skeleton REST flow, token-free setup, blueprint-preset replay, and the paperdoll-is-an-edit trap. The only sub-check it drops is the exact *"costs ~1 generation"* figure on `cheap-animation`, where claude and codex named the cheapest route but declined to state a precise credit price without verifying it (deepseek stated it and scored 2/2) — a strict-scoring artifact, not a routing miss.
- **The official docs get ~58%.** They carry the full MCP inventory, so mcp/docs nails pure MCP tool-selection — hex tiles, characters, the GUI kit, objects, fonts, isometric tiles, sidescroller tilesets, balance — but score **0** on every REST-only route (backgrounds, the `generate-image-v2` sheets/grids, the skeleton flow), near-0 on setup, get mixed credit on blueprint replay, and mostly fall for the paperdoll trap. (Adding those five pure-MCP routes actually lifted the docs' average over the earlier 12-scenario snapshot — the skill's edge concentrates precisely where the inventory is silent.)
- **No skill is mostly lost (~24%).** With nothing but the request, the agent rarely lands on the exact PixelLab tool; it writes a plausible plan around a generic or wrong one. Even the safety floor — declining the unsafe browser-session reuse in `refuse-internal-endpoint` — is no longer a clean sweep once codex and deepseek are included, though it stays the highest-scoring row for the weak arms.
- **Scoring is strict and dry.** Nothing executes (no PixelLab credits, no API calls), and a scenario passes only when the response identifies the correct route — the exact tool/endpoint, or (for the setup, cost, and fallback questions) the documented mode and policy — with no partial credit for a plausible-sounding plan that names the wrong tool. Two scenarios partly score skill-defined conventions the official docs don't carry — the blueprint preset file (`replay-blueprint`) and the `PIXELLAB_SECRET` env-var name (`setup-codex`); they mirror what a skill user actually has on hand, and they are part of why those rows floor the non-skill arms.
- **Efficiency:** Pip's up-front context (~7.5k) is slightly leaner than injecting the docs (~7.7k), but it reads a reference on demand when a task needs one (11 of these 20 scenarios do), which brings the per-scenario average to ~9.1k (median ~8.7k, max ~12.6k — see the context table). The routing accuracy costs more context than the docs on reference-needing tasks; the refactor cut the base `SKILL.md` roughly a third versus Pip (Old v0.5.0).

This snapshot is `claude`, `codex`, and `deepseek-v4-pro`, 1 rep each (the `full` preset). Routing correctness aggregates cleanly across agents — it is simply whether the right tool was named — and the three agents agree closely (Pip 98–100%, docs 55–62%, no-skill 20–30%); session input-token counts are **not** comparable across agents, so read those within an agent. Headline percentages are the mean of per-scenario check rates (each scenario weighted equally), averaged across the three agents; pooling every check instead shifts the weaker arms a few points (e.g. no-skill ~29% pooled vs ~24% averaged in this snapshot). The two strategies are complementary, not rivals — the skill treats the official docs as the authoritative MCP inventory; see [Official MCP Docs vs Pip Skill](tools/pixellab-mcp-docs-vs-pip-skill.md).

## Reproduce

No agent CLIs or secret needed (the PixelLab MCP Docs arm fetches the official doc once). The Pip and git columns are deterministic; regenerate the **context size** table with:

```powershell
python dev-tools/skill_benchmark.py --static --variants pre-kiss-yagni-refactor,mcp-docs,vanilla
```

Agent routing (needs an agent CLI; nondeterministic, but still **dry — no PixelLab credits**). Budget real time: cells run sequentially at ~30 s median (some take 2–3 min), so the full preset's 240 cells took **~2 hours** for the published 1-rep run — and every extra rep adds that again, so keep `--reps` at 1–2. This writes results to `.local/bench/<stamp>/` — read its `SUMMARY.md`:

```powershell
python dev-tools/skill_benchmark.py --agents claude --variants pre-kiss-yagni-refactor,mcp-docs,vanilla `
  --scenarios route-hex-tiles,route-character,route-gui-modular --reps 2
```

To refresh the tables in *this* report, run the **full** scenario set (no `--scenarios` filter) with `--report docs/pixellab-pip-benchmark.md`, or use the `full` preset below — a partial `--scenarios` run with `--report` would shrink the published routing table to only those rows.

On Windows, [`dev-tools/run-skill-benchmark.ps1`](../dev-tools/run-skill-benchmark.ps1) wraps the common runs. Its **`full`** preset runs every agent and every variant (dry, no credits — expect ~2 hours per rep) and refreshes this report automatically. Full methodology, arms, and flags are in [Developer → Skill Benchmark: Routing and Cost](developer.md#skill-benchmark-routing-and-cost).

## Method and caveats

- **Agent session only.** Tokens/cost are the agent's, not PixelLab credits.
- **Route selection, not live outcomes.** A passing check means the response named the correct route; it does not measure whether a live call — parameters, job polling, error recovery — would succeed. Execution is out of scope by design (no credits spent).
- **The MCP Docs arm is the pasted-docs tip, not a connected MCP server.** A live-connected client exposes the real tool schemas in-session, which likely helps pure MCP tool-selection beyond injected markdown; that column measures the "include this link in your prompts" strategy specifically. REST-only routes are unaffected — there is no tool schema to see.
- **The 20 scenarios are hand-picked routing cases** verified against the official docs, not a measured distribution of real user requests.
- **Static tokens are chars/4 estimates**, not a tokenizer count; treat them as relative, not absolute.
- **Session input-token counts are not comparable across agents.** claude, codex, and opencode meter usage very differently (codex/opencode figures run much larger), so read that column *within* an agent, not across them. The context-size table above is the apples-to-apples efficiency comparison.
- **The routing snapshot is nondeterministic.** Agent runs vary; the context table is the reproducible part. Rerun with more agents/reps (or the `full` preset) for your own numbers. The generation note above records when it was last measured and with which agents/reps.
- **Fairness controls:** each variant runs in an isolated workspace outside the repo (so no stray `CLAUDE.md`/`AGENTS.md` leaks in), variants interleave within each rep, medians are reported, and routing is scored by fixed regexes rather than a model.
- **The PixelLab MCP Docs content is fetched live** from `api.pixellab.ai/mcp/docs` at run time and saved alongside the results for audit.
